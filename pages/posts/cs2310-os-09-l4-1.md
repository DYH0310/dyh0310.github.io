# Lecture 4-1 CPU 调度

- 来源：`L4-1-CPU调度-2.pptx.pdf`
- 目标：理解 CPU 调度的目标、评价指标和经典算法，知道不同算法分别解决什么问题、牺牲什么东西。

## 本讲主线

进程已经有了，接下来就必须回答：  
当多个进程都想运行时，CPU 应该先给谁？

这就是调度问题。调度不是单纯追求“公平”，而是在利用率、吞吐量、响应时间、等待时间之间做折中。

## 知识点 1：CPU 执行与 I/O 等待交替出现

- 大多数进程并不是一直纯计算，而是计算一段、等待 I/O 一段，再继续计算。
- 因此操作系统要尽量在一个进程等待时，安排另一个进程运行。
- 调度的基础前提，就是系统里同时存在多个可切换任务。

例子：

- 编译程序往往长时间占用 CPU，属于 CPU-bound。
- 文本编辑器或浏览器经常等待用户输入或磁盘/网络数据，更接近 I/O-bound。

## 知识点 2：什么时候需要进行 CPU 调度

- 进程从运行态进入等待态时。
- 进程从运行态回到就绪态时，例如时间片用完。
- 进程从等待态变成就绪态时，例如 I/O 完成。
- 进程终止时。

例子：

- 一个进程发起磁盘读请求后阻塞，调度器必须立刻挑一个新进程接替 CPU。
- 时钟中断发现时间片耗尽，也可能触发重新调度。

## 知识点 3：抢占式与非抢占式调度

- 非抢占式调度中，进程一旦拿到 CPU，通常要等它主动阻塞或结束才会让出。
- 抢占式调度允许操作系统强制收回 CPU。
- 抢占能提升响应性，但也增加上下文切换和竞争复杂度。

例子：

- 后台长任务如果运行在非抢占系统上，前台短任务可能要等很久。
- 图形界面系统通常需要抢占式调度，否则界面容易卡死。

## 知识点 4：调度程序（dispatcher）负责把“选择”变成“切换”

- 调度器决定选谁。
- 调度程序负责保存当前上下文、恢复目标上下文、切到用户态并跳到合适位置执行。
- 切换本身的代价称为调度延迟或上下文切换开销。

例子：

- 如果时间片设置得极小，虽然响应看起来更及时，但频繁切换会显著增加系统内耗。

## 知识点 5：调度评价指标决定算法设计方向

- CPU 利用率：希望 CPU 尽量忙。
- 吞吐量：单位时间完成多少任务。
- 周转时间：一个任务从提交到完成经历多久。
- 等待时间：任务在就绪队列里总共等了多久。
- 响应时间：从提交请求到第一次得到响应经历多久。

例子：

- 交互式系统更看重响应时间。
- 批处理系统更看重吞吐量和平均周转时间。

## 知识点 6：不同业务场景的目标并不相同

- 前台交互任务通常更关心响应快。
- 后台批处理任务通常更关心整体完成效率。
- I/O 密集型任务和 CPU 密集型任务，也往往适合不同对待。

例子：

- 用户点击按钮后希望立刻响应，这要求调度偏向短、交互型任务。
- 夜间离线报表任务不在乎首响应时间，更在乎总处理效率。

## 知识点 7：FCFS 先来先服务调度

- 按进入就绪队列的先后顺序调度。
- 优点是简单、实现容易。
- 缺点是容易出现“护航效应”：短任务被长任务堵在后面。

例子：

- 如果一个需要 30ms 的长任务先到，后面两个各需 1ms 的短任务也得跟着等，整体平均等待时间会很差。

## 知识点 8：SJF 最短作业优先

- 优先调度预计 CPU 执行时间最短的任务。
- 理论上可获得最优的平均等待时间或平均周转时间。
- 但难点是：系统通常不知道一个任务未来到底会跑多久。

例子：

- 在一组短任务和一个长任务同时等待时，先做短任务，通常能让平均完成时间更好看。

## 知识点 9：预测下次 CPU burst 是 SJF 的现实近似

- 系统可以根据历史行为估计未来 CPU 执行长度。
- 常见方法是指数平均，把最近一次和历史平均结合起来。
- 这是“用过去推未来”的工程化折中。

例子：

- 如果某个进程过去总是短小、频繁 I/O，那么系统会倾向于认为它下一次 CPU burst 也不会太长。

## 知识点 10：优先级调度

- 每个进程赋予一个优先级，优先调度高优先级任务。
- 可以是抢占式，也可以是非抢占式。
- 问题在于低优先级任务可能长期得不到运行机会，出现饥饿。

例子：

- 实时输入处理任务通常优先级高于后台日志压缩任务。

## 知识点 11：轮转调度（RR）适合时间共享系统

- 就绪队列中的每个进程按顺序获得一个时间片。
- 时间片到就切换给下一个进程。
- 它兼顾了公平性和交互响应，是通用分时系统的经典算法。

例子：

- 三个终端用户同时登录一台服务器，每人轮流得到短暂 CPU 时间，因此都感觉系统在“及时响应自己”。

## 知识点 12：时间片长短是 RR 的关键权衡

- 时间片太大，RR 会退化得接近 FCFS。
- 时间片太小，响应快了，但上下文切换开销急剧上升。
- 因此时间片长度必须结合硬件速度和工作负载选择。

例子：

- 如果时间片只有 1ms，而一次切换就要几十微秒，系统会把大量时间浪费在切换上。

## 知识点 13：多级队列调度按任务类别分层

- 不同类型任务进入不同队列，例如前台交互队列、后台批处理队列。
- 每个队列内部可以用不同算法。
- 队列之间可以采用固定优先级或时间比例分配。

例子：

- 交互式任务所在队列始终优先于批处理队列，因此用户操作响应更及时。

## 知识点 14：多级反馈队列（MLFQ）更灵活

- 进程可根据行为在不同优先级队列之间移动。
- 常见策略是：新任务先给高优先级；如果长期占用 CPU，就逐步下调。
- 它试图用“行为反馈”来近似区分交互任务与长任务。

例子：

- 一个新启动的编辑器进程起初响应很快；如果后来开始长时间后台计算，可能被降到较低优先级队列。

## 知识点 15：优先级反转是多任务系统中的真实风险

- 高优先级任务等待低优先级任务持有的资源时，会出现优先级反转。
- 更糟的是，中优先级任务可能不断抢占低优先级任务，导致高优先级任务更久拿不到资源。

例子：

- 高优先级线程要拿一把锁，锁却被低优先级线程持有；此时如果中优先级线程一直运行，低优先级线程反而没机会释放锁。

## 知识点 16：优先级反转的典型解决方法

- 优先级继承：低优先级持锁者临时继承高优先级。
- 优先级上限协议：使用资源前按预设上限提升优先级。
- 目标都是尽快让资源持有者运行并释放关键资源。

例子：

- 如果低优先级线程持有高优先级线程急需的锁，系统可临时把低优先级线程提到高优先级运行。

## 这讲结束后应该形成的认知

- 调度算法没有绝对最优，只有是否适合特定目标。
- 经典算法的真正价值不只是考试会算等待时间，而是理解“系统到底在优化什么、牺牲什么”。
