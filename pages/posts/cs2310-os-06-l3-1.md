# Lecture 3-1 线程

- 来源：`L3-1-线程-2.pptx.pdf`
- 目标：理解线程为何出现、线程和进程的差异、常见线程模型，以及多线程编程中的主要支持机制与问题。

## 本讲主线

这一讲的核心问题是：  
既然进程已经能并发，为什么还要再引入线程？

简短答案是：进程太“重”，而很多并发任务其实共享同一地址空间和资源，把它们拆成线程能更高效、更灵活。

## 知识点 1：线程是 CPU 调度的更细粒度执行单元

- 线程是进程内部的一条执行流。
- 一个进程至少有一个线程，也可以有多个线程并发执行。
- 同一进程内的线程共享代码段、数据段、堆和打开文件，但各自拥有独立的程序计数器、寄存器集合和栈。

例子：

- 浏览器可以让一个线程负责网络接收，一个线程负责页面渲染，一个线程负责脚本执行。
- 文档编辑器可以一边自动保存，一边响应用户键盘输入。

## 知识点 2：为什么需要线程

- 有些任务天然属于同一应用内部的不同活动，不值得拆成多个重量级进程。
- 线程共享地址空间，通信和数据共享更方便。
- 在线程之间切换，通常比进程切换成本更低。

例子：

- 视频播放器中，解码线程与音频播放线程需要共享同一段缓冲区；如果分成独立进程，协调成本会更高。

## 知识点 3：多线程的主要优点

- 响应性更好：一个线程阻塞时，别的线程还能继续工作。
- 资源共享更直接：线程天然共享进程资源。
- 经济性更好：线程创建和切换开销通常小于进程。
- 可扩展性更强：多核机器上可以把不同线程分布到不同核心执行。

例子：

- 浏览器下载大图片时，如果只有单线程，界面可能卡住；有了独立 UI 线程，界面仍可响应用户操作。

## 知识点 4：多核系统让多线程价值进一步放大

- 在单核上，多线程更多体现为“并发交替”。
- 在多核上，不同线程可以真正并行执行。
- 因此多核硬件的发展直接推动了多线程模型和并行编程的普及。

例子：

- 四核 CPU 上，4 个计算线程可能同时运行，而不是轮流抢一个 CPU。

## 知识点 5：Amdahl 定律说明并行加速有上限

- 如果程序中仍有一部分必须串行执行，那么无论增加多少核心，总加速比都存在理论上限。
- 串行部分越大，可获得的并行收益越受限制。
- 这提醒我们：线程多不等于一定快，关键在于并行比例和同步开销。

例子：

- 一个程序 75% 可并行、25% 必须串行，即使不断加核，收益也不会无限增长。
- 如果某程序 40% 是串行部分，那么无论核心数多大，理论加速比上限都是有限的。

## 知识点 6：并行有数据并行和任务并行两种典型形式

- 数据并行：把同样的操作作用在不同数据块上。
- 任务并行：把不同任务分配给不同线程执行。
- 真实应用常常混合两种方式。

例子：

- 对一张大图的不同区域做相同滤波，属于数据并行。
- 一个线程负责接收请求、一个线程负责数据库查询、一个线程负责日志写入，属于任务并行。

## 知识点 7：进程与线程的边界

- 进程强调资源拥有和隔离。
- 线程强调执行流拆分和并发执行。
- 线程依附于进程存在，不能脱离进程单独拥有地址空间。

例子：

- 两个浏览器标签页如果是两个进程，崩溃隔离更强。
- 一个标签页内部的脚本执行和渲染任务更适合用线程组织。

## 知识点 8：线程有用户级与内核级两大实现方向

- 用户级线程由线程库在用户空间管理，内核未必感知每一个线程。
- 内核级线程由操作系统直接管理和调度。
- 两者的性能、阻塞行为和调度能力差异很大。

例子：

- 用户级线程切换可非常快，因为不一定需要陷入内核。
- 但如果一个用户级线程发起阻塞系统调用，整个进程可能一起被阻塞。

## 知识点 9：多线程模型是理解线程实现的关键

### 9.1 多对一模型

- 多个用户线程映射到一个内核线程。
- 优点是管理简单、切换快。
- 缺点是一个线程阻塞可能拖住整个进程，而且无法利用多核并行。

例子：

- 用户库在一个进程里维护多个协作式线程，但内核只看到一个执行实体。

### 9.2 一对一模型

- 每个用户线程对应一个内核线程。
- 优点是并发能力强，阻塞影响范围小，可利用多核。
- 缺点是创建和管理成本更高。

例子：

- 主线程阻塞等待磁盘时，另一个工作线程仍可在其他核心上继续执行。

### 9.3 多对多 / 双层模型

- 多个用户线程复用若干内核线程。
- 目标是在灵活性和内核资源开销之间折中。
- 双层模型允许部分线程直接绑定到底层内核线程。

例子：

- 大量轻量用户线程由运行时调度到有限数量的内核线程上执行，类似现代某些语言运行时的协程/线程池调度思想。

## 知识点 10：线程库是程序员实际接触线程的入口

- 线程库负责线程创建、退出、同步、属性设置等操作。
- 常见接口包括 Pthreads、Windows 线程 API，以及更高层的并行框架。

例子：

- 使用 `pthread_create()` 创建线程，是 POSIX 线程编程的经典入口。
- Windows 下线程创建和同步通常使用 Win32 线程接口。

## 知识点 11：线程池是高频任务处理中的重要工程手段

- 与其来一个任务就创建一个新线程，不如预先创建一组工作线程重复使用。
- 线程池可以降低创建/销毁线程的开销，并帮助限制并发度。
- 这是服务器、消息系统、后台任务执行器中的常见设计。

例子：

- Web 服务器接到请求后，把请求丢给线程池中的空闲工作线程处理，而不是每次请求都临时新建线程。

## 知识点 12：OpenMP 与 TBB 代表更高层的并行编程方式

- OpenMP 适合通过编译指示快速把循环或代码段并行化。
- TBB 更强调任务调度和更高级的并行抽象。
- 这些工具让程序员不必手工管理每一个线程。

例子：

- 在一个循环前加并行指示，编译器和运行时就可能帮你把不同迭代分发给多个线程。

## 知识点 13：多线程编程带来的典型问题

### 13.1 信号处理

- 多线程环境下，信号应该由哪个线程处理并不总是直观。
- 程序需要明确线程级信号屏蔽和处理策略。

例子：

- 一个进程收到终止信号时，可能需要专门的管理线程统一处理退出流程。

### 13.2 线程撤销

- 强行终止一个线程可能让锁、内存和共享状态处于不一致状态。
- 因此线程取消通常要区分异步取消和延迟取消，并设置清理动作。

例子：

- 工作线程正在更新共享链表时被粗暴取消，可能把链表结构破坏掉。

### 13.3 线程本地存储

- 某些数据虽然逻辑上属于线程，但又不适合放在普通局部变量生命周期里。
- 线程本地存储让每个线程都有自己独立的一份变量实例。

例子：

- 每个线程维护自己的错误码、副本缓存或统计计数器，互不干扰。

### 13.4 调度程序激活

- 这是为协调用户线程库和内核线程而提出的一种机制。
- 核心思想是让内核把关键调度事件通知到用户级运行时。

例子：

- 当内核线程阻塞或可运行数量变化时，用户线程库可据此调整映射关系。

## 这讲结束后应该形成的认知

- 线程把“资源拥有者”和“执行流”这两个概念分开了。
- 它是现代并发和并行系统的基础，但线程越多并不自动代表更高性能。
- 真正重要的是：线程模型、共享方式、同步成本和底层调度能力是否匹配。
