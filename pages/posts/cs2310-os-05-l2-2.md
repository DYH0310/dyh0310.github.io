# Lecture 2-2 进程间通信

- 来源：`L2-2-进程间通信-1.pptx.pdf`
- 目标：理解多个进程为什么要通信，以及共享内存、消息传递、管道、套接字、RPC 等常见通信方式的基本思想。

## 本讲主线

进程天然是相互隔离的，但真实系统里，很多任务必须协作完成。  
因此本讲的核心问题是：在“隔离”与“协作”之间，操作系统如何提供安全、可控的通信机制？

## 知识点 1：为什么进程之间需要通信

- 有些进程是彼此独立的，但很多进程需要交换数据、协同推进任务。
- 进程协作可以提升并行性、提高模块化程度、增强信息共享能力。
- 操作系统必须在不破坏隔离性的前提下，提供跨进程通信机制。

例子：

- 浏览器进程可能和 GPU 进程协作完成页面渲染。
- Web 服务进程和数据库进程需要交换查询请求与结果。

## 知识点 2：共享内存是最快的一类 IPC 思路

- 共享内存允许多个进程把同一段物理内存映射到各自地址空间。
- 建立共享区通常需要内核参与，但建立之后的数据读写可以在用户态直接完成。
- 它的优点是高效，缺点是需要程序员自己处理同步。

例子：

- 生产者进程把数据写到共享缓冲区，消费者进程直接从同一块共享区读走。
- 视频处理程序可能用共享内存传递大块图像帧，避免频繁拷贝。

## 知识点 3：共享内存适合大数据交换，但同步问题更突出

- 共享内存本身只解决“看见同一份数据”，不解决“何时可以读写”。
- 如果多个进程同时改同一块共享区域，就会出现竞争条件。
- 因此共享内存常和信号量、互斥锁、条件变量等同步机制一起使用。

例子：

- 生产者刚写到一半，消费者就开始读取，可能得到不完整消息。
- 两个写者同时修改共享队列头尾指针，会导致缓冲区结构损坏。

## 知识点 4：生产者-消费者是共享内存通信的经典模型

- 生产者负责放入数据，消费者负责取出数据。
- 核心问题包括：缓冲区不能满时还继续写，也不能空时还继续读。
- 因此需要同时处理互斥和同步顺序。

例子：

- 日志采集进程写入共享缓冲区，分析进程从缓冲区读取并处理。
- 网卡接收线程把数据包放入环形缓冲区，协议栈线程再把它们取走。

## 知识点 5：POSIX 共享内存是标准化接口实例

- POSIX 提供了共享内存对象和映射接口，用于跨进程建立共享区。
- 典型步骤包括：创建共享对象、设置大小、映射到地址空间、读写数据、解除映射和清理对象。
- 这体现了“内核建立共享关系，用户态直接访问数据”的设计思路。

例子：

- 一个进程用标准接口创建共享内存并写入配置，另一个进程映射同一对象后直接读取。

## 知识点 6：消息传递是另一条 IPC 主线

- 与共享内存不同，消息传递强调通过发送/接收原语交换离散消息。
- 它更像“通过内核或中间机制传递信封”，而不是“共同写一块白板”。
- 消息传递通常更安全、更模块化，但数据复制和内核参与可能带来额外开销。

例子：

- 客户端进程把请求消息发给服务器进程，服务器处理后再回消息。
- 微内核系统常把服务设计成消息交互形式。

## 知识点 7：消息传递有多种设计维度

- 直接通信还是间接通信：是明确指定对方进程，还是通过邮箱/端口中转。
- 阻塞还是非阻塞：发送/接收是否要等待对方或缓冲条件满足。
- 缓冲策略：零容量、有限容量、无限容量。

例子：

- 阻塞接收像“打电话等对方接起”；非阻塞接收像“看看邮箱有没有新邮件”。
- 邮箱机制允许多个发送者都把消息发到同一个逻辑端口。

## 知识点 8：客户机/服务器模型是操作系统通信的重要应用场景

- 很多系统服务本质上都是客户机/服务器模型。
- 客户端发起请求，服务器接收请求、处理请求并返回结果。
- IPC 机制决定了这种交互是本地发生还是跨网络发生。

例子：

- 文件客户端请求文件服务器读取文件。
- 浏览器请求本地 DNS 缓存服务解析域名。

## 知识点 9：套接字把通信抽象为端点

- Socket 是网络通信中最常见的抽象之一。
- 它把通信描述为“两个端点之间的数据流或报文交换”。
- 套接字可用于同一机器内的本地通信，也可用于跨机器网络通信。

例子：

- Web 客户端通过 TCP socket 连接 Web 服务器并发送 HTTP 请求。
- 本机两个服务也可以用 UNIX domain socket 高效通信。

## 知识点 10：RPC 试图把远程调用伪装成本地函数调用

- 远程过程调用（RPC）希望让程序员像调用本地函数一样调用远程服务。
- 其背后需要参数编组、网络传输、结果返回和错误处理。
- RPC 简化了编程模型，但底层仍然是消息传递。

例子：

- 客户端代码写成 `result = query(user_id)`，但实际上这可能触发了一次跨机器请求。

## 知识点 11：普通管道适合有亲缘关系进程的单向数据流

- 管道可以把一个进程的输出连接到另一个进程的输入。
- 普通管道通常用于父子进程或同一祖先进程派生出的进程之间。
- 它天然适合流式处理，但不适合复杂多方通信。

例子：

- Shell 中的 `ls | grep txt` 就是典型管道用法。
- 父进程创建子进程后，用管道把输入数据传给子进程处理。

## 知识点 12：命名管道扩展了管道的适用范围

- 命名管道（FIFO）有名字，可被无亲缘关系的进程通过文件系统路径访问。
- 它保留了“像流一样传输数据”的特点，但通信双方不必来自同一个父进程。

例子：

- 两个独立启动的本地服务可以通过命名管道进行命令和结果传输。

## 知识点 13：不同 IPC 机制的典型适用场景

- 共享内存：大块数据、高性能、本地通信，但同步复杂。
- 消息传递：模块边界清晰、适合服务式设计，但开销通常更高。
- 管道：简单、适合线性数据流。
- 套接字：适合网络化、分布式和本机服务通信。
- RPC：适合把分布式调用封装成服务接口。

例子：

- 图像处理流水线中，原始帧适合共享内存，控制命令适合消息。
- Shell 命令链更适合管道，而 Web 服务调用更适合套接字或 RPC。

## 这讲结束后应该形成的认知

- 进程间通信没有“唯一最优解”，只有“适合当前协作模式的机制”。
- 真正的工程判断标准通常包括：性能、同步复杂度、编程便利性、隔离性和可扩展性。
