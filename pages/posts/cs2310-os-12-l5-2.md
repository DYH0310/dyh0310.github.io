# Lecture 5-2 进程同步实例

- 来源：`L5-2-同步例子-2.pptx.pdf`
- 目标：通过经典问题和主流平台接口，把上一讲的同步原语落到具体模式与实际 API 上。

## 本讲主线

上一讲讲的是同步机制本身。  
这一讲讲的是：这些机制在经典问题和实际系统里如何组合使用。

## 知识点 1：有界缓冲问题是“互斥 + 计数同步”的标准模板

- 缓冲区大小有限，因此生产者不能无限写，消费者也不能在空缓冲区上读。
- 这个问题常用三个信号量建模：
- `mutex` 负责互斥访问缓冲区。
- `empty` 表示剩余空槽数。
- `full` 表示已有数据项数。

例子：

- 生产者先 `wait(empty)` 再 `wait(mutex)`，写完后 `signal(mutex)` 和 `signal(full)`。
- 消费者先 `wait(full)` 再 `wait(mutex)`，读完后 `signal(mutex)` 和 `signal(empty)`。

## 知识点 2：读者-作者问题强调“共享读、独占写”

- 多个读者可以同时读共享对象。
- 作者一旦写入，就必须获得独占访问权。
- 这个问题比简单互斥更复杂，因为它不是“只能一个进”，而是“读可以并发，写必须独占”。

例子：

- 数据库配置表允许多个查询线程同时读取。
- 但更新配置时，必须阻止任何读者和其他写者同时进入。

## 知识点 3：第一类读者-作者问题偏向读者优先

- 目标是：只要当前没有作者已经获得访问权，读者就不应被无谓阻塞。
- 常用 `readcount` 统计当前正在读的读者数。
- 第一个读者负责挡住作者，最后一个读者负责放开作者。

例子：

- 第一个进入的读者执行 `wait(wrt)`，防止作者进入。
- 最后一个离开的读者执行 `signal(wrt)`，允许作者继续。

## 知识点 4：第二类读者-作者问题偏向作者优先

- 目标是：一旦作者已经在等待，就不应再让新读者不断插队。
- 这类方案通常会额外引入控制变量或额外信号量，限制新读者进入。
- 它牺牲部分读者吞吐，换取作者不被长期饿死。

例子：

- 当一个作者就绪后，可以先关住新的读者入口，等现有读者退场后尽快安排作者写入。

## 知识点 5：哲学家就餐问题揭示了“互斥不等于无死锁”

- 每根筷子可看成一个互斥资源。
- 哲学家需要同时拿到左右两根筷子才能吃饭。
- 如果每个人都先拿左边，再等右边，就可能形成环路等待，从而死锁。

例子：

- 5 个哲学家同时拿起左筷子后，所有右筷子都被别人占着，于是谁都吃不到，系统永久僵住。

## 知识点 6：哲学家问题的不同改法体现不同代价

- 随机退让可以降低死锁概率，但不能从理论上彻底消除问题。
- 引入额外互斥保护“拿筷子流程”可以避免死锁，但会显著降低并发度。
- 更优雅的方案通常需要破坏循环等待条件或限制最多同时尝试拿筷子的人数。

例子：

- 用一个全局 `mutex` 保护拿筷子流程，确实能避免死锁，但可能退化成一次只让一个哲学家尝试进餐。

## 知识点 7：Windows 同步对象强调“signaled / nonsignaled”状态

- Windows 中很多同步对象都可处于 signaled 或 nonsignaled 状态。
- signaled 表示资源可用或条件满足，获取时不会阻塞。
- nonsignaled 表示当前不可用，线程获取时会被阻塞。

例子：

- 一个事件对象被置为 signaled 后，等待它的线程可以继续运行。
- 一个互斥对象处于被占用状态时，对应线程获取它会等待。

## 知识点 8：Linux 同步体现了从原子整数到互斥锁的分层

- 现代 Linux 是可抢占内核，因此同步问题非常真实。
- Linux 提供原子整数、互斥锁、信号量等机制。
- 原子整数适合非常小粒度的不可分割计数更新；互斥锁适合保护临界区。

例子：

- `atomic_t counter` 可用于实现不会被中断打断的计数增减。
- `mutex_lock()` / `mutex_unlock()` 用于保护共享链表或关键内核数据结构。

## 知识点 9：Pthreads 把常用同步原语标准化给用户态程序

- Pthreads 为普通应用开发提供了跨平台风格较统一的同步接口。
- 典型原语包括：互斥锁、条件变量、读写锁、信号量。
- 这使应用程序能直接使用成熟同步机制，而不用自己发明底层协议。

例子：

- 多线程服务器可用 `pthread_mutex_t` 保护共享队列，用条件变量在队列空时睡眠等待。

## 知识点 10：Pthreads 互斥锁是最常见的用户态同步起点

- 使用前需初始化。
- 进入临界区前 `pthread_mutex_lock()`。
- 离开临界区后 `pthread_mutex_unlock()`。
- 如果锁不可用，调用线程会阻塞等待。

例子：

- 多个线程共同更新一个哈希表时，可把“查找并插入/修改”的关键片段放在互斥锁保护下。

## 知识点 11：Pthreads 信号量区分命名与无名

- 命名信号量有系统范围的名字，可被无亲缘关系进程共享。
- 无名信号量通常用于同一进程内线程之间，或放在共享内存中供相关进程使用。
- 这种区分反映了“同步作用域”不同。

例子：

- 两个独立进程需要同步某项本地服务资源时，更适合命名信号量。
- 一个线程池内部协调空闲工作槽位时，更适合无名信号量或条件变量。

## 知识点 12：经典同步问题的真正学习价值

- 不是背代码模板，而是学会识别共享状态、资源约束、等待条件和潜在死锁。
- 只要能把问题抽象成“谁共享什么、谁等待什么、谁必须互斥”，就能开始设计同步方案。

例子：

- 订单系统里“库存扣减”本质上像临界区问题。
- 缓存刷新线程和读线程的关系，本质上和读者-作者问题类似。

## 这讲结束后应该形成的认知

- 经典同步问题不是历史包袱，而是现代工程问题的浓缩模型。
- 你在真实系统里看到的大多数并发 Bug，最终都能在“有界缓冲、读写共享、资源竞争、死锁风险”这些模板里找到影子。
