# Lecture 1-2 操作系统结构

- 来源：`L1-2-操作系统结构-3.pptx.pdf`
- 目标：理解操作系统向上提供哪些服务，向下如何与硬件交互，以及内核本身有哪些典型结构。

## 本讲主线

这一讲要回答三个问题：

1. 应用程序是通过什么接口使用操作系统的？
2. 用户态和内核态是如何切换的？
3. 操作系统内部可以组织成哪些结构，它们各有什么优缺点？

## 知识点 1：操作系统提供的基本服务

- 操作系统提供程序执行、I/O 操作、文件系统操作、通信、错误检测、资源分配、统计和保护等服务。
- 这些服务让应用不必直接操控硬件，也不必自行维护复杂的共享策略。
- 操作系统服务既面向单个程序，也面向整个系统。

例子：

- 文本编辑器保存文件时，依赖的是文件系统服务。
- 下载工具访问网络时，依赖的是通信服务。
- 多个程序共享内存和 CPU 时，依赖的是资源分配服务。

## 知识点 2：用户接口并不只有图形界面

- 用户与操作系统交互的常见方式包括命令行界面、图形用户界面和脚本接口。
- 命令行界面的核心是 Shell，它负责解释用户输入的命令并启动相应程序。
- 图形界面把文件、窗口、菜单等抽象可视化，但底层仍会落到系统调用和驱动交互。

例子：

- 在命令行输入 `ls`，Shell 会创建进程执行相应程序，并把结果输出到终端。
- 在图形界面双击文件，本质上仍是由系统找到关联程序并启动对应进程。

## 知识点 3：API 与系统调用不是一回事

- API 是程序员调用的编程接口，例如 C 标准库、POSIX 接口。
- 系统调用是用户程序请求内核服务的底层入口。
- API 可能直接封装系统调用，也可能在用户态先做一层缓冲、格式化或错误处理。

例子：

- `printf()` 是库函数，不一定立刻触发系统调用；缓冲区满了或显式刷新时，才可能调用 `write()`。
- `fopen()` 最终可能会调用 `open()`，但对程序员来说看到的是更方便的标准库接口。

## 知识点 4：系统调用是用户态进入内核态的正规通道

- 用户程序不能随意执行特权操作，否则系统将失去隔离性。
- 当程序需要打开文件、创建进程、分配内存、发送网络数据时，必须通过系统调用请求内核代办。
- 系统调用通常通过软中断、陷入指令或专门的调用门机制实现。

例子：

- 用户程序不能自己改页表，但可以通过 `mmap()` 请求内核建立映射。
- 用户程序不能直接访问网卡寄存器，但可以通过 `send()` 把数据交给内核网络栈。

## 知识点 5：系统调用参数传递与调用分类

- 系统调用除了“调用哪个服务”，还必须把参数安全传给内核。
- 常见传参方式包括寄存器传参、内存块传参、栈传参。
- 系统调用可分为几大类：进程控制、文件管理、设备管理、信息维护、通信、保护等。

例子：

- `open(path, flags, mode)` 需要把路径、打开方式和权限信息传给内核。
- `fork()` 属于进程控制类调用。
- `read(fd, buf, size)` 同时涉及文件描述符、缓冲区地址和读取长度。

## 知识点 6：用户态与内核态是保护机制的核心

- 现代 CPU 通常把执行环境划分为不同特权级，至少区分用户态和内核态。
- 用户态程序权限受限，不能执行敏感指令，不能直接访问受保护资源。
- 内核态拥有更高权限，负责执行设备控制、内存管理和中断处理等任务。

例子：

- 用户程序可以进行普通算术和分支操作，但不能直接关闭中断。
- 只有内核态代码才可以修改页表、设置时钟中断、访问设备控制寄存器。

## 知识点 7：从用户态切换到内核态的三种典型方式

- 系统调用：程序主动请求内核服务。
- 异常：例如缺页、除零、非法指令等，CPU 被动转入内核处理。
- 中断：通常来自外部设备或时钟，打断当前执行流进入内核。

例子：

- 调用 `read()` 是主动陷入内核。
- 程序访问一个尚未装入内存的页面，会产生缺页异常。
- 网卡收到数据包后触发中断，操作系统进入网络中断处理流程。

## 知识点 8：异常与中断处理依赖硬件支持

- 硬件需要提供中断向量、保存现场、切换特权级、转入处理程序等机制。
- 内核收到控制权后，要先保存当前上下文，再根据中断号或异常号选择处理例程。
- 处理完成后，还需要恢复现场并返回到被打断的执行位置，或切换到别的任务。

例子：

- 键盘中断到来时，CPU 根据中断向量找到对应处理程序，再由驱动读取键盘控制器数据。
- 缺页异常发生时，内核会检查访问是否合法，必要时把页面从磁盘换入内存。

## 知识点 9：操作系统设计目标

- 用户目标通常包括方便、易学、可靠、快。
- 系统目标通常包括高效、易维护、易扩展、健壮。
- 很多系统结构选择，本质上是在性能、可维护性和安全性之间做折中。

例子：

- 微内核结构更强调模块化和可靠性，但消息传递开销可能更大。
- 单体内核结构路径更短，性能更好，但内部耦合往往更强。

## 知识点 10：典型的操作系统结构

### 10.1 简单结构 / 单体结构

- 许多早期系统把大量功能都放在同一个内核地址空间中。
- 优点是性能好、调用路径短。
- 缺点是模块边界弱，调试和维护成本高。

例子：

- 早期 UNIX 可以视为典型的单体内核风格。

### 10.2 分层结构

- 系统按层次组织，上层只依赖下层提供的服务。
- 优点是清晰、便于验证和维护。
- 缺点是实际划分层次往往不容易，层间额外开销也可能增加。

例子：

- 可以把“设备管理 -> 内存管理 -> 进程管理 -> 用户接口”理解为一种逻辑分层。

### 10.3 微内核结构

- 微内核尽量只保留最核心机制，如地址空间管理、线程调度、进程间通信。
- 文件系统、设备驱动等尽量放到用户态服务中。
- 优点是可靠、可扩展、组件隔离更强。
- 缺点是服务之间频繁消息传递可能影响性能。

例子：

- 如果文件系统服务崩溃，在微内核里理论上可以只重启该服务，而不一定拖垮整个内核。

### 10.4 模块化/混合结构

- 现代操作系统常采用“单体内核 + 模块化”的折中方式。
- 核心仍在内核态，但通过模块接口降低耦合。
- 这比纯微内核更容易获得性能，也比纯单体更利于扩展。

例子：

- Linux 允许通过可加载内核模块插入驱动程序。
- Windows 和 macOS 也大量采用混合式设计，而不是完全纯粹的某一种结构。

## 知识点 11：操作系统调试与观测

- 操作系统调试比普通应用更难，因为错误可能导致整机失控或无输出。
- 常见手段包括日志、内核调试器、跟踪工具、崩溃转储、性能计数器等。
- 观测能力对系统设计非常重要，很多工程问题不是“实现不了”，而是“看不见问题在哪”。

例子：

- 调度异常时，可以借助跟踪工具查看某个线程为何长时间拿不到 CPU。
- 驱动崩溃时，可以通过 panic 日志或 dump 分析调用栈。

## 知识点 12：系统引导（Boot）是从硬件到操作系统的接力过程

- 机器上电后，CPU 并不会自动进入完整操作系统，而是先执行固件中的引导代码。
- 引导程序负责初始化基本硬件、定位内核映像、把内核装入内存并转交控制权。
- 操作系统启动后，再继续完成设备初始化、文件系统挂载、服务启动等工作。

例子：

- PC 上常见流程是：固件 -> bootloader -> 内核 -> init/systemd -> 用户空间服务。

## 知识点 13：ISA 与特权指令为操作系统提供硬件基础

- 指令集体系结构不仅规定算术和访存指令，也规定特权相关机制。
- 用户级指令供普通程序使用；系统级指令用于控制中断、页表、特权级和设备等。
- 没有特权指令和模式切换机制，就很难构建现代操作系统的保护边界。

例子：

- 用户程序可以执行加法指令，但不能直接执行“装载页表基址”这类系统级操作。
- 操作系统通过特权指令设置时钟中断，才能实现时间片调度。

## 这讲结束后应该形成的认知

- 操作系统不是一个“函数库”，而是一个受硬件保护、通过系统调用向外提供服务的执行环境。
- “接口、特权级、结构”是理解后续所有系统章节的关键底座。
