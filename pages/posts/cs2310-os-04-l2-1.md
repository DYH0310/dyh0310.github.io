# Lecture 2-1 进程

- 来源：`L2-1-进程-3.pptx.pdf`
- 目标：理解“程序如何变成可被操作系统调度和管理的执行实体”，以及围绕进程产生的一整套管理机制。

## 本讲主线

这一讲的核心问题是：一个程序装进内存后，为什么还不够？为什么操作系统一定要引入“进程”这个概念？

答案是：因为程序一旦开始执行，就会涉及寄存器、内存映射、打开文件、运行状态、调度队列和生命周期管理。这些信息必须被操作系统统一记录和控制。

## 知识点 1：程序与进程不是同一个概念

- 程序是静态的，是存放在磁盘上的指令和数据集合。
- 进程是动态的，是程序的一次执行过程。
- 同一个程序可以对应多个不同进程，每个进程都有自己的运行现场和资源状态。

例子：

- 磁盘上的 `python.exe` 是程序；同时开 3 个 Python 脚本，就是 3 个不同进程。
- 浏览器安装包是程序；你打开浏览器后，真正被调度和管理的是浏览器进程。

## 知识点 2：为什么需要进程

- 如果 CPU 只顺序执行一个程序，那么管理很简单，但资源利用率很低。
- 现实中，程序会频繁等待 I/O，如果不切走，CPU 就会空闲。
- 操作系统通过让多个进程交替运行，在一个 CPU 上制造“并发”，提高整体利用率。

例子：

- 文本编辑器等待磁盘保存时，CPU 可以转去执行音乐播放器。
- 下载程序等待网络数据时，CPU 可以去运行编译任务。

## 知识点 3：进程切换的本质是保存和恢复执行现场

- 只修改程序计数器 PC 远远不够，因为进程执行状态不仅包括下一条指令位置。
- 还必须保存通用寄存器、栈指针、页表信息、状态字等上下文。
- 进程切换是“当前进程现场保存到某处，再把另一个进程现场恢复回来”的过程。

例子：

- 进程 A 用寄存器保存了循环变量，进程 B 用同一组物理寄存器做别的事；如果不先保存 A 的寄存器值，A 继续运行时状态就会错乱。

## 知识点 4：PCB 是操作系统管理进程的核心数据结构

- PCB（Process Control Block）用于记录进程的关键状态。
- 典型信息包括：进程标识符、进程状态、程序计数器、CPU 寄存器、调度信息、内存管理信息、I/O 状态和打开文件信息。
- 可以把 PCB 理解为“操作系统眼中的进程身份证 + 管理档案”。

例子：

- 当时钟中断到来时，内核会把当前寄存器内容写入当前进程 PCB，然后再切到别的 PCB。
- `ps` 命令之所以能显示进程信息，本质上依赖内核维护的进程描述数据。

## 知识点 5：进程的基本状态与状态迁移

- 常见状态包括：新建、就绪、运行、等待/阻塞、终止。
- 就绪表示“除了 CPU 之外其他条件都满足”；阻塞表示“还在等待某个事件”。
- 状态转换反映的是进程和 CPU/I/O 的关系变化。

例子：

- 进程请求磁盘读数据后会从运行态进入阻塞态。
- 磁盘完成读操作后，进程从阻塞态回到就绪态。
- 调度器选中它时，它才从就绪态变成运行态。

## 知识点 6：进程的内存布局体现了操作系统的抽象能力

- 一个典型进程地址空间常包括代码段、数据段、堆和栈。
- 代码段存放程序指令；数据段存放全局/静态数据；堆用于动态分配；栈用于函数调用。
- 这套地址空间布局让程序员可以用统一方式理解内存，而不必直接操心物理内存位置。

例子：

- `malloc()` 申请的对象通常来自堆。
- 函数局部变量和返回地址通常在栈上。
- 全局变量 `int g = 3;` 通常位于数据段。

## 知识点 7：调度队列和调度程序决定“谁先运行”

- 就绪队列保存当前可运行但尚未获得 CPU 的进程。
- 设备队列保存等待特定 I/O 设备的进程。
- 短程调度器从就绪队列中选择一个进程交给 CPU。
- 调度程序（dispatcher）负责真正完成上下文切换、模式切换和控制转移。

例子：

- 10 个进程都已经在内存中，但同一时刻单核 CPU 只能运行 1 个，其他 9 个留在就绪队列里等待。

## 知识点 8：进程创建体现了“父子关系”和资源继承

- 一个进程可以创建新的子进程。
- 子进程通常会继承父进程的部分执行环境，例如打开文件、环境变量等。
- UNIX 中典型做法是 `fork()` 创建，再通过 `exec()` 装入新程序。

例子：

- Shell 读到一条命令后，会创建子进程执行该命令。
- 父进程和子进程在 `fork()` 之后会从同一位置继续执行，但返回值不同。

## 知识点 9：fork 是理解 UNIX 进程模型的关键

- `fork()` 会复制当前进程，形成一个父进程和一个子进程。
- 从语义上看，父子进程都有各自独立的地址空间和执行流。
- 从实现上看，现代系统通常不会立即完整复制所有物理页，而是结合写时复制优化开销。

例子：

- `pid = fork();`
- 如果 `pid == 0`，当前执行流位于子进程。
- 如果 `pid > 0`，当前执行流位于父进程，并且返回值是子进程 PID。

## 知识点 10：进程终止不只是“程序结束”

- 进程终止可能因为正常完成、调用 `exit()`、出现异常错误、被其他进程终止等。
- 进程退出后，内核还要负责回收其占用的大部分资源。
- 但内核通常会暂时保留一小部分退出信息，供父进程查询子进程退出状态。

例子：

- 命令执行完毕后，Shell 需要知道它是成功退出还是报错退出。
- 程序访问非法内存导致段错误，也会触发异常终止流程。

## 知识点 11：僵尸进程说明“退出”和“回收”是两回事

- 子进程退出后，如果父进程还没有读取它的结束状态，这个子进程就可能处于僵尸状态。
- 僵尸进程不再运行，但内核仍保留其少量信息以便父进程回收。
- 如果父进程长期不回收，大量僵尸会占据进程表项。

例子：

- 父进程不断创建子进程但从不调用 `wait()`/`waitpid()`，系统里就会累积僵尸进程。

## 知识点 12：wait / waitpid 体现了父进程对生命周期的管理

- `wait()` 用于阻塞等待某个子进程结束。
- `waitpid()` 更灵活，可以指定等待某个子进程，也可以选择非阻塞方式查询。
- 这让父进程既能同步子任务，也能及时回收资源。

例子：

- Shell 启动前台命令后，通常会等待该命令完成再继续接收用户输入。
- 服务进程如果不想卡住自己，可以用 `waitpid(..., WNOHANG)` 轮询回收已退出子进程。

## 知识点 13：Shell 是进程创建与管理的经典实例

- Shell 自己也是一个进程。
- 它的工作流程通常是：读取命令、解析命令、创建子进程、在子进程中执行目标程序、父进程等待或继续。
- 因此 Shell 是理解 `fork`、`exec`、`wait` 关系的最好例子之一。

例子：

- 当你输入 `gcc main.c` 时，Shell 不会自己变成 gcc，而是创建一个新进程去执行编译器。

## 知识点 14：多进程并发会引出同步问题

- 一旦多个进程共享某些数据或逻辑资源，光有“能切换”还不够，还要保证执行顺序和结果正确。
- 这就是后续同步与互斥章节要解决的问题。
- 进程管理和同步问题不是分离的，前者负责“让多个任务一起推进”，后者负责“让它们推进得正确”。

例子：

- 两个进程同时把共享变量 `counter` 加 1，如果没有同步保护，最后结果可能只增加 1 而不是 2。

## 这讲结束后应该形成的认知

- 进程是操作系统管理执行活动的基本单位。
- PCB、地址空间、状态机、调度、创建、终止和回收，构成了进程这一章的完整骨架。
- 后续线程、调度、同步、内存管理，其实都是在这个骨架上继续细化。
