# Lecture 5-1 进程同步

- 来源：`L5-1-同步-2.pptx.pdf`
- 目标：理解并发执行为什么会导致错误，以及互斥、同步、原子操作、锁、信号量、管程这些机制分别在解决什么问题。

## 本讲主线

前面我们一直在追求“让多个任务同时推进”。  
这一讲开始回答另一个同样重要的问题：  
多个任务一起推进时，怎样才能保证结果仍然正确？

## 知识点 1：并发访问共享数据会导致竞争条件

- 多个执行实体同时访问并修改同一份共享数据时，最终结果可能依赖执行顺序。
- 这种“结果取决于交错顺序”的现象称为竞争条件。
- 一旦存在竞争条件，程序就可能出现间歇性、难复现的错误。

例子：

- 两个进程都执行 `counter++`，这条语句底层会分成读、加、写三步；如果交错执行，最终可能只增加 1 次。

## 知识点 2：生产者-消费者问题揭示了同步的本质

- 一个进程生产数据，另一个进程消费数据，是最典型的协作并发场景。
- 仅仅共享缓冲区还不够，必须同时控制“不能越界写”“不能越界读”“不能同时乱改共享状态”。
- 因此它同时包含了互斥问题和前后顺序问题。

例子：

- 缓冲区为空时消费者不能读。
- 缓冲区已满时生产者不能继续写。
- 更新 `counter`、`in`、`out` 等共享变量时又必须互斥。

## 知识点 3：临界区是共享数据最脆弱的部分

- 会读写共享变量、更新共享表项、写共享文件的代码段，就是临界区。
- 设计同步机制的核心，就是保证临界区按正确规则进入。
- 临界区之外的代码称为剩余区。

例子：

- 更新队列头尾指针的几行代码通常就是典型临界区。
- 打印日志字符串前后的普通计算不一定是临界区，但写共享日志缓冲区那几步就是。

## 知识点 4：临界区问题的三个正确性要求

- 互斥：任意时刻至多一个进程在临界区。
- 进步：如果临界区空闲且有人想进，就不能无限拖延选择。
- 有限等待：一个进程请求进入后，不能永远被别人无限插队。

例子：

- 只有互斥没有有限等待，可能导致某个线程一直进不去，出现饥饿。
- 只有互斥没有进步，可能在没人进入临界区时仍然谁都进不去。

## 知识点 5：Peterson 算法展示了软件互斥的经典思想

- Peterson 算法适用于两个进程。
- 它通过 `flag[i]` 表示“我想进临界区”，通过 `turn` 表示“如果冲突，先让谁进”。
- 它满足互斥、进步和有限等待，是教材中的经典解法。

例子：

- 进程 P0 想进入时先把 `flag[0] = true`，再把 `turn = 1`，表示“我想进，但如果对方也想进，就先让 P1”。
- 如果此时 P1 也想进，就通过 `turn` 和 `flag` 的组合决定只允许一个先进入。

## 知识点 6：纯软件方案有教育价值，但工程上常依赖硬件支持

- Peterson 算法说明互斥可以通过读写普通变量实现。
- 但真实系统中，现代处理器存在乱序执行、多核缓存等问题，单纯软件协议不够稳健也不够通用。
- 所以操作系统更常依赖硬件原子指令和内存顺序保证。

例子：

- 在现代多核系统里，如果没有适当的内存屏障，两个核心看到共享变量更新的顺序可能不同。

## 知识点 7：内存屏障解决“看见的顺序”和“执行的顺序”不一致问题

- 现代 CPU 和编译器会为了性能重排指令。
- 并发程序关心的不只是“做没做”，还关心“别人何时能按什么顺序看到”。
- 内存屏障强制约束某些读写不能跨过屏障重排。

例子：

- 线程 A 先写数据再写标志位，线程 B 看到标志位后去读数据；如果没有屏障，B 可能先看到标志变了，却没看到真正的数据更新。

## 知识点 8：原子硬件指令是低层同步原语基础

- 常见原子指令包括 test-and-set、compare-and-swap、swap 等。
- 它们能把“读-改-写”作为一个不可分割操作完成。
- 很多锁和无锁结构都是在这些原子指令基础上构建的。

例子：

- test-and-set 可用来实现一个简单自旋锁：谁先把锁位从 0 改到 1，谁进入临界区。

## 知识点 9：互斥锁是最基础的高层互斥工具

- 互斥锁把“进入临界区”抽象为获取锁，把“离开临界区”抽象为释放锁。
- 只要程序遵守同一把锁保护同一份共享数据，就能获得清晰的临界区边界。
- 互斥锁适合保护短小、明确的共享临界资源。

例子：

- 更新全局计数器前 `lock()`，更新后 `unlock()`，就能避免多个线程同时改写。

## 知识点 10：自旋与阻塞是锁实现的两种不同等待策略

- 自旋：拿不到锁时一直忙等，适合等待时间非常短、上下文切换成本高的场景。
- 阻塞：拿不到锁就睡眠，适合等待时间可能较长的场景。
- 系统会根据临界区长度、运行上下文和调度环境选择合适策略。

例子：

- 中断上下文中不能睡眠，常用自旋。
- 用户态线程等待一个长临界区时，更适合阻塞锁。

## 知识点 11：信号量把“资源数量”和“等待队列”结合起来

- 信号量是一个整数同步原语，典型操作是 `wait(P)` 和 `signal(V)`。
- `wait` 表示尝试占用一个资源单位；若不可用则等待。
- `signal` 表示释放一个资源单位，并可能唤醒等待者。
- 与单纯互斥锁不同，信号量天然适合描述“有多个同类资源”或“事件到来”。

例子：

- `mutex = 1` 的二值信号量可实现互斥。
- `empty = n, full = 0` 常用来表示有界缓冲区里空槽和已填充槽的数量。

## 知识点 12：阻塞型信号量比忙等更适合一般操作系统

- 如果每个等待者都忙等，会浪费 CPU。
- 阻塞型信号量会把等待进程挂起，放入等待链表。
- `signal()` 时从等待链表中唤醒一个进程，让 CPU 做真正有用的工作。

例子：

- 生产者发现缓冲区满时，不是循环空转检查，而是睡眠，等消费者释放空间后再被唤醒。

## 知识点 13：管程把同步提升到更高层抽象

- 管程把共享数据和对共享数据的操作封装在一起。
- 管程保证任意时刻只有一个执行实体活跃在其内部。
- 程序员不必在每个操作外层手工散布锁逻辑，结构更清晰。

例子：

- 一个有界缓冲区可以被封装成 `put()` 和 `get()` 两个管程操作，外部只调用接口，不直接操作内部计数和队列。

## 知识点 14：条件变量用于“等待某种状态成立”

- 互斥锁解决“同一时刻谁能进入”，条件变量解决“什么时候应该继续”。
- `wait()` 通常会在睡眠前释放锁，被唤醒后再重新获取锁。
- `signal()` 表示某个条件可能已满足，应通知等待者重新检查。

例子：

- 消费者在 `Count == 0` 时对 `notEmpty` 调用 `wait()`。
- 生产者放入数据后对 `notEmpty` 调用 `signal()`，提醒消费者“现在可能有东西可读了”。

## 知识点 15：用管程解决生产者-消费者更接近工程写法

- 进入 `put()` 或 `get()` 时先拿锁。
- 条件不满足时在对应条件变量上等待。
- 修改共享状态后发出适当通知，再释放锁。
- 这种写法比手工散布多个信号量更容易维护。

例子：

- `while (Count == n) notFull.wait(&lock);`
- `while (Count == 0) notEmpty.wait(&lock);`
- 这类模式在实际并发代码中极其常见。

## 知识点 16：同步机制有明显层次结构

- 最底层是硬件原子操作与内存顺序保证。
- 中间层是锁和信号量。
- 更高层是管程、条件变量等结构化抽象。
- 真正的系统设计通常是分层使用，而不是只靠某一种工具。

例子：

- 一个高级语言里的 `synchronized`、monitor 或 channel，底层最终仍可能建立在原子指令和锁之上。

## 这讲结束后应该形成的认知

- 同步的目标不是“让程序慢下来”，而是“在并发前提下仍保持正确性”。
- 互斥锁、信号量、条件变量、管程解决的问题不完全相同，关键是知道何时该用哪一种。
