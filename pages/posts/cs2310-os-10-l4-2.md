# Lecture 4-2 CPU 调度进阶

- 来源：`L4-2-CPU调度-2.pptx.pdf`
- 目标：把调度问题从单 CPU 推进到线程、多处理器、多核和实时系统，并了解 Linux、Windows、Solaris 等真实系统中的典型做法。

## 本讲主线

上一讲主要回答“单处理器上怎么调度进程”。  
这一讲继续回答三个更接近真实系统的问题：

1. 当调度单位变成线程时，会发生什么变化？
2. 当机器变成多处理器/多核时，调度还要额外考虑什么？
3. 实际操作系统如何把理论调度算法落成工程实现？

## 知识点 1：有线程时，调度对象通常变成线程而不是进程

- 如果系统支持内核级线程，内核调度器看到的是线程。
- 线程之间共享进程资源，但在 CPU 竞争上往往各自独立。
- 因此调度粒度从“进程”进一步细化到“线程”。

例子：

- 同一个浏览器进程里的渲染线程和网络线程，可能分别被调度到不同 CPU 上执行。

## 知识点 2：PCS 与 SCS 区分了不同层次的竞争范围

- PCS（Process-Contention Scope）表示竞争 CPU 发生在同一进程内部线程之间，常见于用户级线程库调度。
- SCS（System-Contention Scope）表示竞争 CPU 发生在系统内所有线程之间，由内核调度。
- 这个划分反映了“线程到底先由谁选，再交给谁执行”。

例子：

- 一个用户级线程库先在本进程内部挑出一个用户线程，这属于 PCS。
- 内核再决定哪个内核线程真正上某个物理 CPU，这属于 SCS。

## 知识点 3：Pthreads 暴露了线程竞争范围配置能力

- Pthreads 线程属性中可以设置竞争范围策略。
- `PTHREAD_SCOPE_PROCESS` 对应偏向进程内部竞争。
- `PTHREAD_SCOPE_SYSTEM` 对应系统范围竞争。
- 这说明线程库接口不仅用于“创建线程”，也用于指定线程如何参与调度。

例子：

- 程序可通过线程属性控制新线程更像“用户线程映射到轻量级进程”，还是“直接由系统统一调度”的风格。

## 知识点 4：多处理器调度首先要解决负载如何分配

- 当系统有多个 CPU 时，除了“先给谁”之外，还要回答“给哪颗 CPU”。
- 如果多个处理器功能相同，就主要考虑负载平衡和共享数据开销。
- 如果某些 I/O 设备与特定处理器耦合更强，还会带来额外的绑定倾向。

例子：

- 某个任务频繁访问与处理器 A 更接近的设备或缓存，把它迁到处理器 B 可能反而变慢。

## 知识点 5：非对称多处理与对称多处理

- 非对称多处理（AMP）中，一个主处理器承担调度和系统管理，其他处理器主要执行用户任务。
- 对称多处理（SMP）中，每个处理器都可参与调度和内核活动。
- SMP 灵活且常见，但需要更谨慎地处理共享数据结构并发访问。

例子：

- 在 SMP 中，两个 CPU 可能同时从同一就绪队列挑任务，因此必须避免选中同一个线程两次。

## 知识点 6：处理器亲和性是缓存友好性的体现

- 线程最近运行在哪个处理器上，会影响其缓存命中情况。
- 如果频繁迁移处理器，原缓存失效、新缓存重填，代价可能很高。
- 因此系统常尽量让线程留在熟悉它工作集的处理器上，这就是处理器亲和性。

例子：

- 一个不断扫描大数组的线程如果一直留在同一个核心上，缓存局部性通常更好；频繁迁核会导致重复冷启动。

## 知识点 7：软亲和性、硬亲和性与 NUMA

- 软亲和性表示“尽量不迁移”，但必要时仍可移动。
- 硬亲和性表示显式限制线程只能在某些处理器上运行。
- 在 NUMA 架构中，不同 CPU 到不同内存区域的访问代价不同，亲和性就更重要。

例子：

- 某数据库线程如果主要访问本 NUMA 节点内存，把它固定在对应 CPU 组上通常更高效。

## 知识点 8：负载平衡与亲和性经常相互拉扯

- 负载平衡希望所有处理器都别太闲，也别太忙。
- 亲和性希望线程尽量别乱迁，保住缓存局部性。
- 系统需要在“均衡”与“局部性”之间做动态折中。

例子：

- 一个 CPU 明显过载、另一个 CPU 空闲时，系统可能会牺牲亲和性把任务迁过去，这叫 push migration 或 pull migration。

## 知识点 9：多核处理器带来新的硬件层次问题

- 多核处理器把多个核心集成到同一芯片上，可能共享部分缓存或内存通路。
- 这让“多个核心”不再只是多份独立 CPU，还涉及共享层次缓存和内存停顿问题。
- 调度器如果理解这些拓扑关系，可以做出更好的放置决策。

例子：

- 两个高度竞争缓存带宽的线程放在同一芯片邻近核心上，可能互相拖慢；把它们分散开反而更好。

## 知识点 10：实时调度追求的是截止期与确定性

- 普通分时调度强调平均性能，而实时调度强调任务是否能在规定时间前完成。
- 实时系统通常分为硬实时和软实时。
- 硬实时任务错过期限可能不可接受；软实时任务错过期限则主要影响质量。

例子：

- 飞控控制回路更接近硬实时。
- 音视频播放更接近软实时，偶尔延迟会卡顿，但不一定导致系统失效。

## 知识点 11：实时调度中的几个经典策略

### 11.1 优先级实时调度

- 为实时任务赋予固定或动态优先级。
- 高优先级任务优先获得 CPU。

例子：

- 电机控制线程优先级高于状态日志线程。

### 11.2 单调速率调度（RMS）

- 周期越短的任务优先级越高。
- 适合周期性任务集合。

例子：

- 每 5ms 执行一次的传感器采样线程优先级可高于每 50ms 执行一次的状态汇总线程。

### 11.3 最早截止期限优先（EDF）

- 按当前最早截止时间排序。
- 截止时间最近的任务先运行。

例子：

- 两个任务都已就绪时，如果任务 A 的截止时刻比任务 B 更近，就优先调度 A。

## 知识点 12：POSIX 实时调度给出标准化接口

- POSIX 为实时调度策略和优先级提供标准接口。
- 这让程序员可以在兼容系统上显式声明调度需求，而不只是被动接受默认时间共享策略。

例子：

- 工业控制程序可以为关键线程设置实时优先级和策略，以减少普通后台任务的干扰。

## 知识点 13：Linux 调度体现了“公平 + 类别优先”的思路

- Linux 是抢占式系统，并同时支持普通任务和实时任务。
- 普通任务调度强调长期公平，典型思路是让每个任务获得与权重相匹配的处理器份额。
- 讲义中的练习提到 `vruntime`，这正是 Linux 公平调度思想里的关键量：谁的“已享受运行时间”更少，谁更应该被优先照顾。

例子：

- 两个 CPU 密集型任务中，`nice` 值更高权重更低的任务，其 `vruntime` 增长得更快，因此会更早被认为“已经运行够多了”。
- 一个 I/O 密集型任务因为常常睡眠，回来时 `vruntime` 往往相对较小，更容易被尽快响应。

## 知识点 14：Windows 调度体现了“优先级驱动 + 动态调整”

- Windows 采用抢占式、优先级驱动调度。
- 线程优先级受进程优先级类和线程相对优先级共同影响。
- 为改善交互体验，系统通常还会对某些线程进行动态优先级提升。

例子：

- 前台窗口相关线程可能因为交互需要而获得更积极的调度机会。

## 知识点 15：Solaris 等系统展示了类调度器思想

- 不同系统往往把任务分为不同调度类，例如时间共享、交互、实时等。
- 每个类内部再有自己的优先级和策略。
- 这说明真实系统很少只靠单一算法一把抓到底。

例子：

- 实时类任务和普通时间共享类任务，即使都在“调度器”里，也通常遵循不同优先级规则。

## 知识点 16：调度算法评估不能只靠直觉

- 一种算法好不好，必须先定义评估准则。
- 讲义给出三种典型方式：分析评估、仿真和实际实现。
- 不同方法适用于不同阶段：理论分析、原型验证、真实系统观测。

例子：

- 在已知一组任务到达时间和执行长度时，可以先用确定性模型比较 FCFS、SJF、RR 的平均等待时间。
- 在复杂负载下，可以通过仿真程序生成统计数据。
- 最后还可以把算法放进真实系统或实验内核中实际测量。

## 这讲结束后应该形成的认知

- 调度一旦进入线程、多核和真实系统，就不再是单纯的“算平均等待时间”。
- 真正的系统调度需要同时考虑：竞争范围、缓存局部性、负载平衡、实时需求和工程实现成本。
