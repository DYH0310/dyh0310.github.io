# Lecture 3-3 课程回顾

- 来源：`L3-3-课程回顾-1.pptx.pdf`
- 目标：把前面关于 CPU、进程、线程的内容串成一条连续逻辑链，而不是彼此孤立地记忆。

## 本讲主线

这一讲相当于一个小阶段总结，核心是把以下问题串起来：

1. CPU 为什么不能只盯着一个程序一直跑？
2. 操作系统为什么需要进程？
3. 进程之上为什么还需要线程？
4. 用户级线程和内核级线程在本质上差在哪？

## 知识点 1：单程序顺序执行不能充分利用 CPU

- 如果只有一个程序顺序跑，那么遇到 I/O 等待时，CPU 会闲下来。
- 这会造成资源浪费，也是引入多道程序和进程管理的直接动机。

例子：

- 程序正在等待磁盘读取数据时，如果没有其他任务可切换，CPU 只能干等。

## 知识点 2：让 CPU “好好运转”需要多进程

- 操作系统通过多进程让不同任务交替使用 CPU。
- 某个进程等待 I/O 时，CPU 可以切给别的就绪进程。
- 这就是并发的最直接收益：把等待时间藏到别人的计算里。

例子：

- 编辑器在保存文件时阻塞，编译器进程可以趁机继续占用 CPU 编译代码。

## 知识点 3：只改 PC 不足以完成切换

- 从一个程序切到另一个程序，不只是换下一条指令地址。
- 还要保存寄存器、栈、内存映射等完整上下文。
- 这就是为什么操作系统必须维护 PCB。

例子：

- 线程/进程被切走前如果不保存栈指针，恢复后连函数返回到哪里都不知道。

## 知识点 4：进程调度是系统并发运行的关键机制

- 有了多个进程后，仍然要解决“谁先运行、运行多久、何时切换”的问题。
- 这就自然引出调度队列、调度器、时间片等概念。

例子：

- 三个就绪进程都想运行，调度器必须选出当前获得 CPU 的那一个。

## 知识点 5：系统调用把进程管理能力暴露给应用

- 应用程序通过 `fork`、`exec`、`wait` 等系统调用参与进程生命周期管理。
- 这说明进程不是只属于内核内部的概念，也与应用开发接口直接相关。

例子：

- Shell 启动新命令时会调用创建进程相关接口，结束后还要等待并回收子进程。

## 知识点 6：多线程是进程内部进一步提升并发性的手段

- 一个进程内部可能同时有多个活动，它们共享地址空间但执行目标不同。
- 把这些活动建模为线程，比拆成多个进程更轻量、更自然。

例子：

- 浏览器既要联网下载，又要显示文字，还要响应点击操作，适合拆成多个线程。

## 知识点 7：线程模型决定了阻塞行为和并行能力

- 如果是用户级线程，内核未必看得到每个线程。
- 如果是内核级线程，内核就能对每个线程分别调度。
- 因此模型不同，系统调用阻塞的影响范围也不同。

例子：

- 用户级线程中，一个阻塞 I/O 可能挂住整个进程。
- 内核级线程中，一个线程等待 I/O 时，另一个线程仍可继续运行。

## 知识点 8：用户级线程切换本质上是运行时保存/恢复上下文

- 线程库需要保存当前线程栈指针和现场，再恢复目标线程现场。
- 这说明“线程切换”不是一个抽象黑箱，而是明确的实现过程。

例子：

- `Yield()` 把当前线程的 `esp` 存入自己的 TCB，再把另一个线程的 `esp` 读回来。

## 知识点 9：这一阶段的知识链条

- CPU 是被争用的核心资源。
- 进程让程序成为可管理的执行实体。
- 调度让多个进程交替推进。
- 线程把进程内部的并发也显式化。
- 用户态/内核态和系统调用把这些机制连接起来。

例子：

- 一个浏览器标签页最终会涉及：进程、线程、系统调用、调度、I/O、同步，这说明前面学的内容其实是同一套系统的不同切面。

## 这讲结束后应该形成的认知

- 回顾课的价值不在于重复，而在于建立因果关系。
- 只要你能把“CPU 利用率 -> 多进程 -> 调度 -> 线程 -> 线程模型”这条链讲清楚，前几讲的骨架就已经搭起来了。
